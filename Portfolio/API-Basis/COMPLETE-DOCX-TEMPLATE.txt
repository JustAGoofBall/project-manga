LEERUITKOMST 1: API'S BOUWEN OP BASIS VAN ENDPOINTS

Ik ben in staat API's te bouwen op basis van duidelijk gedefinieerde endpoints. Dit vereist inzicht in architectuur, request/response cycles, en professionele API design.

WAT IS EEN API?

API staat voor Application Programming Interface. Het is de manier waarop programma's met elkaar communiceren. REST API is het meest gebruikte type, werkend over HTTP met duidelijke endpoints.

In Project Manga communiceren mijn React frontend en Express backend via een REST API. De frontend stuurt requests naar endpoints als /api/anime, /api/auth/login, enz., en ontvangt JSON responses.

APIS BOUWEN BIJ PROJECT MANGA

Ik heb 30+ werkende endpoints gebouwd met een professionele architectuur:

• Models: Database queries (anime, characters, users, ratings, favorites)
• Controllers: Business logica (validatie, autorisatie, data transformatie)
• Routes: URL mapping (GET /api/anime, POST /api/anime/:id, etc.)
• Middleware: Cross-cutting concerns (authenticatie, error handling, logging)

Deze scheiding van concerns maakt de code testbaar, onderhoudbaar en schaalbaar.

REQUEST/RESPONSE CYCLE

{
  "success": true,
  "count": 5,
  "data": [
    {"id": 1, "name": "Naruto"},
    {"id": 2, "name": "One Piece"}

STATUS CODES CORRECT GEBRUIKT:

• 200 OK = Succes voor GET/PUT
• 201 Created = Resource aangemaakt (POST)
• 400 Bad Request = Ongeldige input (validatiefout)
• 401 Unauthorized = Geen authentificatie (geen token)
• 403 Forbidden = Geen autorisatie (user is geen admin)
• 404 Not Found = Resource bestaat niet
• 500 Internal Server Error = Serverfout

ARCHITECTUUR VOOR APIS

MVC Model (Model-View-Controller):

ROUTE → CONTROLLER → MODEL → DATABASE
  ↓        ↓          ↓        ↓
URL     Logica    Queries   Data

1. Route: Ontsluit de URL (/api/anime/:id)
2. Controller: Valideert input, checkt autorisatie, roept model aan
3. Model: Voert database queries uit
4. Database: Slaat data op, retrieves data

Voordelen van deze architectuur:

✓ Testbaar - Elk component apart testbaar
✓ Onderhoudbaar - Weet waar je bugs moet zoeken
✓ Schaalbaar - Makkelijk nieuwe endpoints toevoegen
✓ Beveiligd - Middleware kan validatie/autorisatie toepassen

ERROR HANDLING

Alle endpoints hebben centralized error handling. Als ergens iets fout gaat, wordt een consistent error response gegeven:

{
  "success": false,
  "message": "Descriptive error message",
  "error": "technical details for debugging"
}

Dit zorgt ervoor dat:
• Frontend weet wat er fout ging
• Gebruiker krijgt helpful message
• Developer kan debuggen met technical details

LEERUITKOMST 2: ENDPOINTS VOOR EEN API OPSTELLEN

Ik kan endpoints voor een API opstellen volgens RESTful design principes. Dit vereist begrip van HTTP methods, resource-based URLs, en professionele API design patterns.

RESTFUL DESIGN PRINCIPES

REST staat voor Representational State Transfer. De kernprincipes zijn:

PRINCIPE 1: RESOURCE-BASED URLS

Het belangrijkste: Resources (nouns), niet acties (verbs)!

❌ SLECHT - Action-based URLs:
/api/getAnime
/api/createAnime
/api/updateAnime/1
/api/deleteAnime/1

✓ GOED - Resource-based URLs:
GET    /api/anime       # Get all anime
POST   /api/anime       # Create new anime
GET    /api/anime/1     # Get one anime
PUT    /api/anime/1     # Update anime
DELETE /api/anime/1     # Delete anime

Waarom? Dit is de industrie standaard. Iedereen herkent deze pattern direct.

PRINCIPE 2: HTTP METHODS HEBBEN SEMANTIEK

GET    = Read data (veilig, idempotent) - geen side effects
POST   = Create new resource - return 201 Created
PUT    = Update existing resource - return 200 OK
DELETE = Delete resource - return 200 OK

Door juiste methods te gebruiken wordt je API zelf-documenterend:
• POST /api/anime = Ik ga iets maken
• GET /api/anime = Ik ga iets lezen
• DELETE /api/anime/1 = Ik ga iets verwijderen

PRINCIPE 3: HIERARCHISCHE STRUCTUUR

Resources kunnen genest zijn:

/api/anime                      # Top-level resource
/api/anime/1                    # Specific anime
/api/anime/1/characters         # Sub-resource (characters van anime 1)
/api/anime/1/characters/5       # Specific character in anime 1
/api/anime/1/ratings            # Sub-resource (ratings van anime 1)
/api/anime/1/ratings/10         # Specific rating

Dit maakt relaties duidelijk en logisch.

PRINCIPE 4: QUERY PARAMETERS VOOR FILTERING

Query parameters zijn voor filtering, sorting, pagination - NIET voor resource identificatie:

GET /api/search?q=naruto              # Search voor naruto
GET /api/anime?sort=name&order=asc    # Sort by name
GET /api/anime?limit=10&offset=0      # Pagina 1, 10 items

CONTROLLER IMPLEMENTATIE

Hier een voorbeeld van hoe createAnime controller eruit ziet:

exports.createAnime = async (req, res, next) => {
  try {
    // STAP 1: Validate input
    const validationError = validateAnimeName(req.body.name);
    if (validationError) {
      return res.status(400).json({
        success: false,
        message: validationError
      });
    }

    // STAP 2: Check authorization
    if (!req.user.isAdmin) {
      return res.status(403).json({
        success: false,
        message: 'Admin access required'
      });
    }

    // STAP 3: Create in database
    const animeId = await Anime.create(req.body.name);

    // STAP 4: Return success with 201 status
    res.status(201).json({
      success: true,
      message: 'Anime created successfully',
      data: { id: animeId, name: req.body.name }
    });
  } catch (error) {
    next(error); // Pass to error handler middleware
  }
};

Elke stap is belangrijk en in dezelfde volgorde:
1. Validatie
2. Autorisatie
3. Database operatie
4. Response

RESPONSE FORMAT CONSISTENCY

Alle endpoints gebruiken dezelfde response format. Dit is essentieel!

Succes response:
{
  "success": true,
  "message": "Anime created successfully",
  "data": { "id": 1, "name": "Naruto" },
  "count": 1
}

Error response:
{
  "success": false,
  "message": "Name is required",
  "error": "ValidationError"
}

Door consistent response format krijgt frontend predictable data om mee te werken.

ERROR HANDLING

Niet alle requests slagen. Goede error handling is essentieel:

try {
  const response = await fetch('http://localhost:3000/api/anime/999');
  
  // Check HTTP status
  if (!response.ok) {
    throw new Error(`HTTP error! status: ${response.status}`);
  }
  
  const data = await response.json();
  console.log(data);
  
} catch (error) {
  console.error('Fetch error:', error.message);
  // Toon user-friendly error message
}

COMMON SCENARIOS:

• 401 Unauthorized: Token is invalid/expired → Re-login gebruiker
• 403 Forbidden: User doesn't have permission → Toon "access denied"
• 404 Not Found: Resource doesn't exist → Toon "not found"
• 500 Server Error: Server crash → Toon "something went wrong"

const login = async (email, password) => {

Ik ben in staat API's automatisch te testen met feature tests (integration tests). Dit vereist begrip van Jest, Supertest, test planning, en coverage rapportage.

WAAROM GEAUTOMATISEERDE TESTEN?

Handmatig testen is:
✗ Traag - elke functionaliteit opnieuw testen
✗ Foutgevoelig - iets vergeten
✗ Niet herhaalbaar - inconsistent
✗ Duur - veel tijd kost geld

Automatische testen zijn:
✓ Snel - in seconden compleet testen
✓ Betrouwbaar - altijd hetzelfde
✓ Herhaalbaar - onbeperkt keer runnen
✓ Goedkoop - eenmaal schrijven, regelmatig gratis runnen

TESTING FRAMEWORK STACK

Jest = Test runner
- Testen schrijven
- Assertions checken (expect)
- Coverage rapportage

Supertest = HTTP testing library
- Requests naar API sturen
- Responses testen
- Geïntegreerd met Jest

Combinatie = Integration testing

TEST TYPEN

1. HAPPY PATH TESTS - Alles werkt

Normale flow, alles gaat goed:

it('should create anime when admin', async () => {
  const res = await request(app)
    .post('/api/anime')
    .set('Authorization', `Bearer ${adminToken}`)
    .send({ name: 'New Anime' });
  
  expect(res.status).toBe(201);
  expect(res.body.success).toBe(true);
  expect(res.body.data.id).toBeDefined();
  expect(res.body.data.name).toBe('New Anime');
});

2. ERROR PATH TESTS - Expected failures

Abnormale situaties, moet correct failuren:

it('should fail without admin token', async () => {
  const res = await request(app)
    .post('/api/anime')
    .set('Authorization', `Bearer ${userToken}`)
    .send({ name: 'New Anime' });
  
  expect(res.status).toBe(403);
  expect(res.body.success).toBe(false);
  expect(res.body.message).toContain('Admin');
});

it('should reject invalid input', async () => {
  const res = await request(app)
    .post('/api/anime')
    .set('Authorization', `Bearer ${adminToken}`)
    .send({ name: '' }); // Empty name!
  
  expect(res.status).toBe(400);
  expect(res.body.message).toContain('required');
});

3. AUTHENTICATION TESTS

Register & login flow:

describe('Authentication', () => {
  it('should register new user', async () => {
    const res = await request(app)
      .post('/api/auth/register')
      .send({
        username: 'testuser',
        email: 'test@example.com',
        password: 'password123'
      });
    
    expect(res.status).toBe(201);
    expect(res.body.token).toBeDefined();
    expect(res.body.user.id).toBeDefined();
  });

  it('should fail with duplicate email', async () => {
    // Register eerste user

      .post('/api/auth/register')
      .send({
        username: 'user1',
        email: 'same@example.com',
        password: 'pass123'
      });

    // Try tweede user met zelfde email
    const res = await request(app)
      .post('/api/auth/register')
      .send({
        username: 'user2',
        email: 'same@example.com',
        password: 'pass456'
      });

    expect(res.status).toBe(400);
  });
});
